# Decision Log

> Каждое архитектурное и процессное решение фиксируется здесь.
> Формат: дата, решение, почему, альтернативы, итог.

---

## 2026-04-01 — Выбор методологии разработки

**Решение:** BMAD-METHOD (AI-driven agile framework) + spec-driven подход

**Почему:** BMAD обеспечивает структурированный AI-driven agile процесс: специализированные агенты (PM, Architect, UX, Dev), 34+ workflow, трассируемая цепочка PRD→ADR→OpenAPI→Code. Это гарантирует воспроизводимость, качество архитектурных решений и прозрачность для команды.

**Альтернативы:**
- Ad-hoc промптинг — быстрее на старте, но теряется контроль качества на масштабе
- Собственный workflow — долго создавать, нет community-backing
- Waterfall — не адаптивен к изменениям требований

**Итог:** BMAD-METHOD как основа процесса, каждый шаг документируется.

## 2026-04-01 — Выбор инструментов разработки

**Решение:** Claude Code + 7 скиллов + 2 MCP + 3 кастомных скилла + GitHub Issues

**Почему:** Автоматизация рутинных операций (ревью, тесты, диаграммы, трекинг) позволяет сфокусироваться на продуктовых решениях. Каждый инструмент закрывает конкретную задачу в pipeline разработки.

**Скиллы:**
| Скилл | Задача в pipeline |
|---|---|
| UI/UX Pro Max | Дизайн-система: стили, палитры, шрифты |
| frontend-design | Качественный UI без generic дефолтов |
| Excalidraw Generator | Архитектурные диаграммы для документации |
| Code Reviewer | Автоматическое ревью после каждой фазы |
| webapp-testing | E2E тесты UI через Playwright |
| frontend-slides | Генерация презентаций из кода |
| skill-creator | Быстрое создание кастомных скиллов |

**Кастомные скиллы:** /log-decision, /create-issue, /screenshot

**MCP:** Playwright (UI тесты + скриншоты), PostgreSQL (запросы к БД при отладке)

**Альтернативы:**
- Минимальный набор (только IDE) — больше ручной работы, медленнее итерации
- Cursor + расширения — меньше возможностей для кастомизации pipeline

**Итог:** Полный тулчейн ставим на этапе подготовки среды.

## 2026-04-01 — Выбор трекинга прогресса

**Решение:** GitHub Issues (milestones + labels) + DECISION_LOG.md + docs/PROCESS.md

**Почему:** Три уровня трекинга для разных аудиторий:
1. GitHub Issues — оперативный трекинг задач, видимый всей команде
2. Decision Log — история инженерных решений с обоснованиями (git blame)
3. PROCESS.md — высокоуровневый нарратив процесса разработки

**Альтернативы:**
- Только TODO.md — не видно внешним участникам
- Linear/Notion — внешний сервис, лишняя зависимость
- Только GitHub Projects — избыточно для текущего масштаба

**Итог:** Тройной трекинг обеспечивает прозрачность на всех уровнях.
